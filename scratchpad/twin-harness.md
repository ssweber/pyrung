# Twin Harness v1

## Summary

- Build a single canonical pyrung program that contains all twin cases plus a shared harness scaffold.
- Use a packed `@named_array(Int, count=32, stride=8)` for case state so each case owns one contiguous DS slice.
- Gate all cases behind a global `TwinArm` bit so the harness can stage writes with execution disabled, then release all cases together.
- Run the source side as a soft PLC from the canonical pyrung program.
- Run the soft reference side from a round-tripped program generated by `pyrung_to_ladder(...)` and `ladder_to_pyrung(...)`, so soft-vs-soft validates export/import behavior instead of only exercising identical in-memory logic.
- Keep the end-to-end suite outside normal pytest discovery and expose it through `make test-twin`.

## Implementation Changes

### Harness memory model

- Reserve `C1` for `TwinArm`.
- Reserve `DS1` for `TwinCaseCount` and `DD1` for `TwinBuild`.
- Define `TwinCase` as a packed named array with fields:
  `Cmd`, `Fired`, `FireLimit`, `Result1`, `Result2`, `Result3`, `Result4`, `ErrorCode`.
- Map `TwinCase` to `DS101..DS356`, giving each case one contiguous 8-word DS slice.
- Define a per-case DS auxiliary window as `@named_array(Int, count=32, stride=8)` with fields `In1..In8`, mapped to `DS501..DS756`.
- Define a per-case C auxiliary window as `@named_array(Bool, count=32, stride=16)` with fields `B1..B16`, mapped to `C101..C612`.
- Define shared timer and counter pools:
  `TwinTimer = Timer.clone("TwinTimer", count=128)` mapped to `T1..T128` / `TD1..TD128`;
  `TwinCounter = Counter.clone("TwinCounter", count=128)` mapped to `CT1..CT128` / `CTD1..CTD128`.
- Give each case fixed helper access to its own core slice, DS aux slice, C aux slice, 4 timer slots, and 4 counter slots.

### Canonical program structure

- Add a first-scan rung that writes `TwinBuild` and `TwinCaseCount`.
- Implement every case with the same gate condition:
  `TwinArm AND TwinCase[i].Cmd == 1 AND (TwinCase[i].FireLimit == 0 OR TwinCase[i].Fired < TwinCase[i].FireLimit)`.
- Keep `Fired += 1` inline in the gated case rung after the case logic so each case remains self-contained.
- Keep case assertions ladder-owned:
  values under test are copied into `Result1..Result4` or the case-owned DS/C auxiliary windows on the scan that matters.
- Do not let the harness read instruction-internal timer, counter, or transient system state directly unless the case snapshots it into owned observation slots.
- Default to inline case rungs; use subroutines only for behavior that specifically depends on `call()` or `return_early()`.

### Case authoring model

- Add a case helper surface that exposes:
  `index`, `core`, `ints`, `bits`, `timer(slot)`, and `counter(slot)`.
- Define one `TwinCaseSpec` per case with:
  `name`, `index`, `fire_limit`, `sample_after_scans`, `inputs`, `expected`, and `compare_rules`.
- Use `sample_after_scans` as the case readiness contract; the harness waits for the maximum required scan count across the selected cases.
- Keep case registration append-only:
  one case module provides the rungs and snapshots,
  one registry entry declares runtime inputs and expected results.

### Endpoint behavior

- `SoftTwinEndpoint` runs `PLC + ClickDataProvider + ClickServer` with a background scan loop at `dt=0.010`.
- `RemoteClickEndpoint` wraps `pyclickplc.ClickClient` and talks only through Modbus reads and writes.
- In soft reference mode:
  build the canonical ladder bundle,
  write `nicknames.csv`,
  regenerate pyrung code with `ladder_to_pyrung(..., nickname_csv=...)`,
  `exec()` the generated source,
  run its `logic` and `mapping` as the reference soft PLC.
- In remote reference mode:
  export the canonical bundle into the per-run artifact directory,
  assume the real PLC is already loaded with that build,
  verify `TwinBuild` and `TwinCaseCount` over Modbus before running.

### Harness flow

- Zero the reserved case windows and helper pools over Modbus with `TwinArm=0`.
- Poll until the reset state is visible.
- Write all selected case inputs, `Cmd`, and `FireLimit` values while `TwinArm=0`.
- Poll until staged values read back correctly.
- Write `TwinArm=1`.
- Capture the source and reference `system.sys.scan_counter` baseline.
- Poll both endpoints until the target scan count is reached.
- Read all case-owned result and observation slots.
- Diff source vs reference and compare both against case expectations using per-tag rules:
  `exact`, `float_tol`, `info_only`.

### Local suite and artifacts

- Put end-to-end tests in root `twin_tests/` so default pytest discovery remains unchanged.
- Add `make test-twin` as:
  `uv run pytest -q -s twin_tests`.
- Keep unit tests for harness pieces in the normal `tests/` tree.
- Write each run to `.pytest_tmp_local/twin/<timestamp>/` with:
  `main.csv`,
  `subroutines/*.csv`,
  `nicknames.csv`,
  `reference_roundtrip.py`,
  `twin_report.md`,
  `twin_report.json`,
  `source_snapshot.json`,
  `reference_snapshot.json`.

### Public interfaces and env vars

- Add `TwinCaseSpec`, `TwinCompareRule`, `TwinEndpointConfig`, and `TwinRunReport`.
- Support:
  `PYRUNG_TWIN_REFERENCE_MODE=soft|remote`,
  `PYRUNG_TWIN_REAL_HOST`,
  `PYRUNG_TWIN_REAL_PORT`,
  `PYRUNG_TWIN_REAL_DEVICE_ID`,
  `PYRUNG_TWIN_SIM_PORT`,
  `PYRUNG_TWIN_CASES`.
- Omit `PYRUNG_TWIN_REFERENCE_PATH` in v1.

## Test Plan

- Unit test packed named-array addressing so case-major DS layout is fixed and documented.
- Unit test helper-pool allocation for DS, C, timer, and counter ownership.
- Unit test build-checksum generation and remote preflight mismatch handling.
- Unit test compare-rule formatting for `exact`, `float_tol`, and `info_only`.
- Unit test reset/write planning for case core windows, DS aux windows, C aux windows, and timer/counter helper pools.
- End-to-end local test: source soft PLC vs round-tripped reference soft PLC with all seed cases enabled in one run.
- End-to-end local test: forced mismatch produces readable markdown and JSON output grouped by case and logical tag.
- End-to-end remote test: skip unless `PYRUNG_TWIN_REAL_HOST` is set; require successful `TwinBuild` preflight before execution.

## Assumptions and Defaults

- Seed cases are:
  `program_structure_subroutine_return`,
  `scan_snapshot_nested_branch`,
  `timer_counter_reset_cycle`,
  `system_points_safe_observe`.
- `Result1..Result4` remain the common observation path; larger cases use the fixed per-case DS/C auxiliary windows.
- No DD or DF auxiliary windows are included in v1.
- The remote PLC is dedicated to the harness and safe to run the canonical exported program.
