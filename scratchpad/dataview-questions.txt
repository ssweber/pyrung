  Design: Import/Export and Multiple Data View Windows for pyrung VS Code Debugger

  pyrung is a Python ladder logic DSL with a VS Code debug extension. The "Data View" is a webview panel
  in the debug sidebar where users watch/force/patch tag values during a debug session. Currently there's
  a single Data View instance with tags added via right-click "Add to Data View".

  What we want to design:

  1. Multiple Data View windows — Users working with large programs want separate views for different
  subsystems (e.g., "Conveyor I/O", "Safety Interlocks", "Timer State"). Each view has its own watched
  tags/groups.
  2. Import/Export — Save a Data View's tag list to a file (JSON or similar), reload it later or share
  with teammates. Think of it like PLC programming tools where you save "watch lists".

  Current architecture:
  - PyrungDataViewProvider is a WebviewViewProvider registered once as pyrung.dataView in package.json
  - It holds _watchedTags (Set), _watchedGroups (Set), _latestTagGroups (from trace events)
  - Tags added via right-click context menu or programmatic addTag()/addGroup() calls
  - Trace updates arrive via updateTrace(tagValues, forces, tagTypes, tagGroups) — filters to only
  relevant tags before posting to webview
  - The webview renders a table with value/type/force columns, group headers, bool buttons, numeric inputs

  Key questions:
  1. Should multiple views be separate VS Code webview panels (floating/dockable) or tabs within a single
  webview? VS Code's WebviewViewProvider gives one sidebar slot — multiple instances likely need
  createWebviewPanel or the WebviewView serializer pattern.
  2. Where should watch lists persist? Workspace .vscode/ folder? A pyrung-views.json at project root? VS
  Code workspace state API?
  3. Should export format include just tag names, or also layout (column widths, collapsed groups,
  ordering)?
  4. How does trace data fan out to multiple views efficiently? Currently updateTrace filters per-view —
  with N views, that's N filter passes per trace event.
  5. Should views be named/renamable? Auto-created from file context (e.g., "Tags from conveyor.py")?
