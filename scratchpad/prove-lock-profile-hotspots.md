Profiled a `pyrung lock` run from a generated project and traced the hot spots back to concrete prove/codegen paths.

## Executive summary

The dominant cost is not the compiled ladder logic itself. The biggest hotspot is the prove wrapper around the compiled kernel, specifically the inline block sync step generated by `prove/kernel.py`.

The profile shows:

- Total runtime: `256.1s`
- BFS steps: `46,044`
- `dict.get()` calls: `320.9M`
- `dict.get()` self time: `57.9s`
- `<block_sync>._step` self time: `85.2s`
- `<block_sync>._step` cumulative time: `147.2s`

## Main finding: block sync dominates

`_compile_inline_step()` in `src/pyrung/core/analysis/prove/kernel.py:25` generates a wrapper that:

1. Copies block-backed tag values from `tags` into block arrays before every scan.
2. Calls the compiled kernel.
3. Flushes block arrays back into `tags` after every scan.

For the sliced lock cluster I inspected, the compiled context still carries `6,061` block-backed tag slots:

- `DS`: `4,500`
- `DH`: `500`
- `DF`: `500`
- `C`: `397`
- `A_AlmCoil`: `100`
- `Cmd_MLk`: `32`
- `Sts_MLk`: `32`

That matches the profile almost exactly:

- `6,061 * 46,044 = 279,072,684`
- Profiled `dict.get()` calls inside the step wrapper: `279,072,086`

The difference is only `598`, so the `dict.get()` storm is effectively explained by block reloads alone.

## Why block sync stays so large

Compaction is controlled by `build_compact_block_maps()` in `src/pyrung/circuitpy/codegen/context.py:375`.

The current rule is coarse:

- If a block is used indirectly anywhere, that block does not get compacted.

In the sliced lock cluster:

- `DS`, `DH`, and `DF` are marked as indirect blocks.
- `C`, `A_AlmCoil`, `Cmd_MLk`, and `Sts_MLk` do compact.

So the prove wrapper still reloads the entire `DS`/`DH`/`DF` universe on every BFS step, even though the lock proof only depends on a much smaller subset.

## Slice breadth is still substantial

The slice is not tiny even for a single lock output:

- `54-55` main rungs
- `33` subroutines
- `1,022` referenced tags
- `229` stateful dimensions
- `45` nondeterministic dimensions
- `23` edge tags
- `42` threshold vectors
- `62` threshold events
- `8` done specs

Representative subroutines still present in the slice include multiple task/state families, mode/state logic, traffic-light logic, validation, and shared helpers. This makes the state-key and hidden-event machinery meaningfully expensive even after identifying block sync as the main issue.

## Secondary hotspots

### State key construction

The profile shows about `32.0s` in state-key construction:

- `_state_key()` in `src/pyrung/core/analysis/prove/__init__.py:319`
- `state_key()` in `src/pyrung/core/analysis/prove/kernel.py:242`

The main sub-costs are:

- `live_edges()`: `17.0s`
- `_extract_state_key()`: `14.8s`
- `_threshold_vector_key()`: `19.3s` cumulative across callers

This is real work, but still clearly second-order behind the wrapper.

### Hidden-event handling

The profile shows about `26.1s` in hidden-event jumping:

- `_maybe_jump_hidden_event()` in `src/pyrung/core/analysis/prove/events.py:390`
- `_resolve_nearest_exact_hidden_event()`: `12.5s`
- `_abstract_threshold_outcomes()`: `7.6s`
- `_scans_until_threshold_event()`: `11.3s`

This makes sense given the slice contents:

- `42` threshold vectors
- `62` threshold events
- several timer/counter abstractions in the state key

### Pre-BFS classification

The one-time setup before BFS still costs `44.5s`:

- `_run_pre_bfs_pipeline()` in `src/pyrung/core/analysis/prove/passes.py:413`
- dominated by `_pass_find_threshold_absorptions()` and `_pass_classify_dimensions()`

This is noticeable, but the BFS wrapper remains the largest opportunity.

## Edge conditions are not the main problem

Rise/fall contacts compile through `_prev.get(...)` in `src/pyrung/circuitpy/codegen/compile/_core.py:337`, with the edge-specific paths at:

- `src/pyrung/circuitpy/codegen/compile/_core.py:410`
- `src/pyrung/circuitpy/codegen/compile/_core.py:421`

But in the sliced kernels I inspected there were only about `25` `_prev.get(...)` sites, versus `6,061` block slot reloads per scan. Edge lookup overhead exists, but it is not the dominant cost in this run.

## Most promising optimization targets

1. Remove or sharply reduce per-scan block reload/flush in the prove wrapper.
2. Improve compaction policy for indirect blocks so one dynamic access does not force full-width `DS`/`DH`/`DF` syncing.
3. Reduce repeated threshold/state-key recomputation.
4. Narrow reachability slices further so unrelated task families do not inflate hidden-event and threshold bookkeeping.

## Notes

One discrepancy is worth preserving here: the current default lock projection partitions into multiple independent clusters, but the profiled run shows a single `_bfs_explore` call. That suggests the run was likely done with an explicit projection or against slightly different local code. The hotspot diagnosis still stands, because the dominant wrapper cost is explained directly by the sliced block-slot count.
